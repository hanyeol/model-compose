# JSON-RPC Client Component

> **Note**: The `json-rpc-client` component is not currently registered in the DSL schema. This document describes a proposed/legacy interface and may not match the running codebase. If you need JSON-RPC today, use `http-client` (for HTTP transport) or `websocket-client` (for WebSocket transport) and encode the JSON-RPC envelope manually in the action body.

The JSON-RPC client component enables invoking remote procedures on external JSON-RPC 2.0 servers. It abstracts the JSON-RPC envelope (`jsonrpc`, `id`, `method`, `params`, `result`, `error`) so workflows only need to declare the method name and parameters. Both HTTP and WebSocket transports are supported.

## Basic Configuration

```yaml
component:
  type: json-rpc-client
  url: https://api.example.com/rpc
  action:
    method: getUser
    params:
      id: ${input.user_id}
    output:
      name: ${response.name}
      email: ${response.email}
```

## Configuration Options

### Component Settings

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `type` | string | **required** | Must be `json-rpc-client` |
| `url` | string | **required** | URL of the JSON-RPC server. Scheme determines the transport: `http(s)://` uses HTTP, `ws(s)://` uses WebSocket |
| `transport` | string | auto | Explicit transport: `http` or `websocket`. When omitted, inferred from `url` scheme |
| `headers` | object | `{}` | HTTP headers included on every request (HTTP transport) or on the handshake (WebSocket transport) |
| `params` | object | `{}` | Default query parameters appended to the connection URL (WebSocket transport only) |
| `timeout` | string | `null` | Maximum time to wait for a single RPC response (e.g. `30s`, `2m`) |
| `ping_interval` | string | `null` | Ping interval for WebSocket connections (e.g. `20s`). WebSocket transport only |
| `ping_timeout` | string | `null` | Ping timeout for WebSocket connections (e.g. `10s`). WebSocket transport only |
| `rate_limit` | object \| string | `null` | Optional rate limit applied to all actions in this component |
| `actions` | array | `[]` | List of JSON-RPC actions |

### Action Configuration

JSON-RPC client actions support the following options:

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `method` | string | **required** | Name of the remote method to invoke (the `method` field in the JSON-RPC request) |
| `params` | object \| array | `null` | Parameters passed as the `params` field. Use an object for named parameters or an array for positional parameters. Omit to send a request with no params |
| `headers` | object | `{}` | Additional headers merged with the component-level headers for this call |
| `timeout` | string | `null` | Per-action timeout that overrides the component-level `timeout` |

## Request and Response Envelope

The component constructs the JSON-RPC 2.0 request envelope automatically:

```json
{
  "jsonrpc": "2.0",
  "id": "<auto-generated>",
  "method": "<action.method>",
  "params": <action.params>
}
```

The `id` field is auto-generated per call (UUID-based for HTTP, monotonically increasing integer for a WebSocket connection) and matched against incoming responses on WebSocket transports. Workflow expressions never need to reference `id` directly.

On success, the server's `result` value is exposed to the workflow as `${response}`. Object fields are accessible via dotted paths (e.g. `${response.balance}`, `${response.items[0].name}`).

## Usage Examples

### Simple HTTP Call

```yaml
component:
  type: json-rpc-client
  url: https://rpc.example.com
  action:
    method: math.add
    params:
      a: ${input.a as number}
      b: ${input.b as number}
    output:
      sum: ${response}
```

### Positional Parameters

```yaml
component:
  type: json-rpc-client
  url: https://rpc.example.com
  action:
    method: math.subtract
    params: [ ${input.a as number}, ${input.b as number} ]
    output:
      result: ${response}
```

### Multiple Actions Component

```yaml
component:
  type: json-rpc-client
  url: https://eth-mainnet.example.com
  headers:
    Authorization: Bearer ${env.RPC_TOKEN}
  actions:
    - id: get-block-number
      method: eth_blockNumber
      output:
        block: ${response}

    - id: get-balance
      method: eth_getBalance
      params: [ ${input.address}, latest ]
      output:
        balance: ${response}

    - id: get-transaction
      method: eth_getTransactionByHash
      params: [ ${input.tx_hash} ]
      output:
        from: ${response.from}
        to: ${response.to}
        value: ${response.value}
```

### WebSocket Transport

```yaml
component:
  type: json-rpc-client
  url: wss://rpc.example.com/ws
  ping_interval: 20s
  ping_timeout: 10s
  headers:
    Authorization: Bearer ${env.RPC_TOKEN}
  action:
    method: chain.getStatus
    output:
      height: ${response.height}
      synced: ${response.synced}
```

When the same component is referenced by multiple actions, the WebSocket connection is reused across calls. The connection is established lazily on the first action invocation.

### Explicit Transport Override

The transport is normally inferred from the URL scheme, but it can be set explicitly when running behind a proxy that rewrites the scheme:

```yaml
component:
  type: json-rpc-client
  url: https://proxy.example.com/upstream/rpc
  transport: websocket
```

### Named Parameters

```yaml
component:
  type: json-rpc-client
  url: https://api.example.com/rpc
  action:
    method: users.create
    params:
      name: ${input.name}
      email: ${input.email}
      active: ${input.active as boolean | true}
    output:
      user_id: ${response.id}
```

## Authentication Patterns

Authentication is delegated to the underlying transport (HTTP headers for HTTP, handshake headers for WebSocket).

### Bearer Token

```yaml
headers:
  Authorization: Bearer ${env.RPC_TOKEN}
```

### API Key

```yaml
headers:
  X-API-Key: ${env.RPC_API_KEY}
```

### Custom Headers

```yaml
headers:
  X-Auth-Token: ${env.RPC_AUTH_TOKEN}
  X-Client-ID: ${env.CLIENT_ID}
```

## Error Handling

A JSON-RPC response with an `error` member is treated as a failed action and stops workflow execution. The error object is propagated with its `code`, `message`, and optional `data` fields preserved.

| Failure source | Treated as |
|----------------|------------|
| HTTP `4xx` / `5xx` status | Transport failure — workflow fails |
| WebSocket disconnect mid-call | Transport failure — workflow fails |
| JSON-RPC `error` member present | Application failure — workflow fails with the server-provided `code` / `message` / `data` |
| RPC response not received within `timeout` | Timeout — workflow fails |

Reserved JSON-RPC error codes (`-32700` Parse error, `-32600` Invalid Request, `-32601` Method not found, `-32602` Invalid params, `-32603` Internal error, `-32000`–`-32099` Server error) are surfaced verbatim from the server; the component does not re-interpret them.

For workflow-level recovery, wrap the action in a workflow with error handling rather than catching errors inside the component.

## Variable Interpolation

JSON-RPC client supports dynamic configuration:

```yaml
component:
  type: json-rpc-client
  url: ${env.RPC_URL}
  action:
    method: ${input.method_name}
    params:
      target: ${input.target}
      count: ${input.count as number}
      verbose: ${input.verbose as boolean | false}
    headers:
      X-Request-ID: ${generate_uuid}
```

## Best Practices

1. **Pick the right transport**: Use HTTP for simple request/response APIs and WebSocket when the server expects a persistent session or many calls in quick succession.
2. **Reuse one component per endpoint**: All actions on a component share the same connection and authentication context; create separate components for distinct endpoints rather than overriding `url` per action.
3. **Environment variables for secrets**: Store API keys and tokens in environment variables, not in YAML.
4. **Set explicit timeouts**: Configure `timeout` for any action whose target method may block, so a stuck server cannot hang the workflow.
5. **Choose params shape deliberately**: Use object form for named parameters (matches most modern APIs); use array form only when the server contract requires positional parameters.
6. **Rate limit external endpoints**: When calling a shared third-party endpoint, set `rate_limit` at the component level to stay within quotas.

## Integration with Workflows

Reference the JSON-RPC client in workflow jobs:

```yaml
workflow:
  jobs:
    - id: fetch-block
      component: eth-rpc
      action: get-block-number

    - id: fetch-balance
      component: eth-rpc
      action: get-balance
      input:
        address: ${input.wallet_address}

    - id: summarize
      component: summarizer
      input:
        block: ${fetch-block.output.block}
        balance: ${fetch-balance.output.balance}
```

## Limitations

The initial implementation is intentionally minimal. The following JSON-RPC 2.0 features are **not supported** yet and will be addressed in follow-up work:

- **Notifications** (requests with no `id`, fire-and-forget): every call expects a response.
- **Batch requests** (array of requests in a single envelope): each action issues exactly one request.
- **Server-initiated subscriptions** (e.g. `eth_subscribe` push streams over WebSocket): WebSocket transport handles request/response only; unsolicited server messages are ignored.

If your use case requires any of the above, use the `http-client` or `websocket-client` component directly and construct the envelope manually until these features land.

## Common Use Cases

- **Blockchain nodes**: Ethereum, Bitcoin Core, Solana, and other JSON-RPC nodes
- **Language servers & debuggers**: LSP and DAP services that speak JSON-RPC
- **Internal microservices**: Backend services that expose JSON-RPC endpoints instead of REST
- **Legacy RPC integrations**: Systems that standardized on JSON-RPC before REST/gRPC adoption
